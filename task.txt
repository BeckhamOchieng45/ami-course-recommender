INTELLIGENT COURSE RECOMMENDATION ENGINE FOR THE AI COACH BOT
Role: Software Engineer (AI Coach Bot &amp; Learning Platform)
Time budget: ~4–8 hours. We respect your time — a focused, well-reasoned submission beats an exhaustive one.
If you run out of time, write down what you&#39;d do next.
Stack: Python or TypeScript (your choice — both are core to our platform). PostgreSQL/SQLite fine for storage.
Use any libraries you like.
1. YOUR TASK
Build a course recommendation service that, given a user, returns the top N recommended courses (default
N=5), each with:
• a ranked position,
• a score (or confidence), and
• a short human-readable reason explaining why this course was recommended for this user.
The &quot;reason&quot; matters: the coach bot will surface it to the participant (&quot;Because you told us you want to improve at
financial planning and finished Intro to Bookkeeping, we suggest…&quot;). So recommendations must be explainable,
not a black box.
1.1 DATA
We provide (or you generate — see §1.2) synthetic data with roughly this shape. Model it however you think is
best; this is a suggested schema, not a mandate.

• courses: course_id, title, topic/category, level (beginner/intermediate/advanced),
skills_taught (tags), duration_mins, prerequisites (optional)
• users: user_id, role, industry, company_size, seniority, stated_goal
• usage_events: user_id, course_id, event_type (started/completed/dropped), progress_pct,
quiz_score, timestamp
• survey_responses: user_id, skill_gaps (tags), goals (tags), preferred_topics (tags),
confidence_by_topic
1.2 GENERATING DATA
If we haven&#39;t shipped you a dataset, write a small script to generate a realistic synthetic one (e.g. ~200
courses, ~1,000 users with usage + survey data). We&#39;d rather see 1,000 well-structured users than a token
handful. Include the generator in your repo.
1.3 REQUIRED CAPABILITIES
1. Blend all three signal sources. A recommendation driven only by usage, or only by the survey, is not
enough. Show how you weight and combine them, and make that weighting legible.
2. Cold-start. A brand-new user has no usage history — only work info + survey. Your engine must still
produce sensible recommendations for them. Explain how you handle the transition from cold-start to
behavior-driven as usage accumulates.
3. Sensible filtering. Don&#39;t recommend a course the user already completed. Respect level/prerequisites
where it makes sense.
4. Explainability. Each recommendation carries a clear, specific reason.
5. Serve it via a small API. e.g. GET /users/{user_id}/recommendations?n=5 returning JSON. A
CLI is acceptable in addition but we want to see an API.
1.4 EXPLICITLY OPTIONAL (BONUS, ONLY IF YOU HAVE TIME)

• Use an LLM to generate the coaching-style message wrapping a recommendation, or to enrich course/user
understanding (embeddings for semantic matching). Tell us why an LLM earns its place vs. plain logic — we
care about that judgment.
• A minimal UI (a single page listing a user&#39;s recommendations).
• Wire up a real PostgreSQL instance rather than in-memory/SQLite.
Do not feel pressure to do all the bonus items. We would much rather see excellent core recommendation logic
than a thin version of everything.

# AMI Course Recommendation Engine — Full Roadmap

**Case study for:** Software Engineer, AI Coach Bot & Learning Platform, African Management Institute (AMI)
**Time budget:** 4–8 focused hours
**Goal:** Build a submission that reads as if it was designed *for AMI specifically*, not a generic recommender adapted to fit the brief.

## PART 1 — COMPANY CONTEXT (keep this open while you build)

### Who AMI is
- Pan-African social enterprise, founded 2014 in Nairobi by Rebecca Harrison (CEO, ex-Reuters correspondent) and Jonathan Cook (Chairman, ex-GIBS Business School director).
- Mission: *"skilled people build thriving businesses, thriving businesses create quality jobs, quality jobs drive prosperity and dignity."*
- Scale: 40,000+ businesses trained, 39+ countries, ~134 employees, ~$8.1M revenue. Mid-sized, real company — not a 5-person startup, not enterprise-scale either. Pragmatic engineering, not over-architected.

- Outcomes: 86% of clients report improved business performance; NPS of 80; internal "impact intelligence" systems track this. **They are an evidence/outcomes-driven org, not a content-consumption org.**

### Their pedagogy — the single most important fact for this project
**70% hands-on experience / 20% learning skills (coaching, peer support) / 10% theory.**
This should influence: how you weight usage signals (practical application > passive viewing), how you write recommendation reasons (emphasize applicability, not just topic match), and how you frame the "measuring success" experiment (behavior change > course completion alone).

### Their learner base
- Spans the *entire* seniority ladder: micro-business owners → SME managers → corporate employees → senior executives.
- Company size skews micro/SME with a corporate/enterprise tail.
- Delivery is mobile-first, sometimes low-bandwidth/low-connectivity — this has real API/architecture implications (see Phase 4).
- Access model is historically free or very low-cost with an optional paid certificate — pricing must not distort recommendation fairness.

### Their programme areas (use these as your course categories, not generic ones)
1. **Entrepreneurship & Business Growth** — bookkeeping, cash flow, pricing, customer service, sales, business planning
2. **Executive, Leadership & Management** — team management, delegation, performance management, organisational/systems change
3. **Workplace Learning & Professional Growth** — communication, productivity, collaboration (aimed at early-career/entry-level staff)
4. Emerging tracks worth including for texture: **AI strategy for senior leaders**, **women's leadership development**

### The theme to thread through everything
Practical, locally-relevant, outcome-measured skill-building, delivered affordably at scale, blending human coaching with digital tools. Every design choice below should trace back to this in one sentence if someone asks "why did you do it that way."

### Phase 1 — Data generator with AMI-flavored realism (60–90 min)

**Courses (~200):**
- [ ] Distribute across the 3–5 AMI programme areas above, not generic categories
- [ ] Levels mapped to AMI's real learner range: micro-entrepreneur / SME manager / senior executive (rename from generic beginner/intermediate/advanced if it helps clarity)
- [ ] `skills_taught` tags should read like AMI's real tool names (e.g. "cash flow forecasting," "delegation frameworks," "customer retention tactics") — specific and practical, not abstract ("finance," "leadership")
- [ ] Add prerequisites sparingly — only where a real learning progression exists (e.g. basic bookkeeping → cash flow forecasting)

**Users (~1,000):**
- [ ] Roles: micro-business owner, SME manager, corporate employee, executive/senior leader
- [ ] Company size distribution skewed toward micro/SME with a smaller corporate tail
- [ ] Industries relevant to the African SME context: retail, agriculture, financial services, manufacturing, professional services, NGO/development
- [ ] `stated_goal` phrased the way AMI's own testimonials read: outcome-oriented ("improve cash flow management," "grow my team," "scale the business") — not abstract ("learn finance")

**The structural trick — do not skip this:**
- [ ] Give every synthetic user a **hidden true-interest variable** (e.g. "cash flow management," "team leadership")
- [ ] Have that hidden variable **probabilistically drive both** their survey answers (goals, skill_gaps, preferred_topics) **and** their usage events (which courses they start/complete, quiz scores), with realistic noise
- [ ] This is what lets you later verify — and show in your sample outputs — that the engine actually *recovers* the user's real interest, rather than just running without crashing

**Usage events — reflect the 70/20/10 model:**
- [ ] Weight `completed` events with high `progress_pct` and high `quiz_score` (practical application) more heavily in your usage signal than `started`-then-`dropped` events
- [ ] Consider a plausible drop-off pattern (some realistic fraction of started courses are dropped, reflecting real mobile/connectivity constraints) — mention this as a deliberate realism choice in the writeup

**Deliverable:** generator script in repo, produces courses.json/csv, users.json/csv, usage_events.json/csv, survey_responses.json/csv (or DB tables).

### Phase 2 — Core scoring engine (90–120 min)

**Architecture — build as pluggable scorers, not one function:**
```
score_component(user, course) -> (score: float, weight: float, reason_fragment: str)
```
Register components in a list/registry. This single decision:
- Pre-answers "how would you add a new signal without rewriting the engine"
- Makes weighting legible (you can log/print each component's contribution)
- Makes the "entry-level users getting advanced courses" debugging question trivial (inspect the breakdown)

**Component 1 — Survey match:**
- [ ] Overlap (Jaccard or weighted overlap) between `goals + skill_gaps + preferred_topics` and course `topic + skills_taught`
- [ ] Weight `goals` and `skill_gaps` higher than `preferred_topics` (stated intent > passive preference)

**Component 2 — Usage-based:**
- [ ] Content similarity to courses the user completed with high progress/quiz_score (tag overlap or simple TF-IDF over `skills_taught`)
- [ ] Cohort popularity fallback: "users with your role/industry/seniority who completed X also completed Y" — this doubles as your cold-start bridge

**Component 3 — Work-context:**
- [ ] Seniority → level relevance (micro-entrepreneur → foundational modules; executive → leadership/change modules)
- [ ] Industry → topic affinity (lightweight heuristic table is fine)

**Cold-start blending — make it a continuous function, not a branch:**
```
usage_confidence = min(1, num_completed_events / K)     # K ≈ 3–5
w_usage_effective = w_usage_base * usage_confidence
# redistribute the freed weight to survey + work-context
```
- [ ] Implement this exact continuous transition — it answers the cold-start requirement *and* preempts "how does this evolve as usage grows" in one design choice

**Filtering (before scoring):**
- [ ] Hard exclude: already completed / in-progress (configurable)
- [ ] Hard exclude: unmet prerequisites
- [ ] Level mismatch: **soft penalty, not hard filter** — decide and document why (a strong match elsewhere can outweigh a level mismatch; hard filtering here is exactly the kind of thing that causes "entry-level users getting advanced courses" bugs)
- [ ] Fairness check: confirm free vs. paid/certificate courses aren't systematically reordered by price — note this explicitly as a deliberate choice consistent with AMI's low-cost-access mission

**Deliverable:** working scoring engine, unit-testable component functions, one aggregate ranking function.

### Phase 3 — Explainability layer (30–45 min)

- [ ] Template reasons off the **highest-contributing component**, not a generic sentence
- [ ] Reflect the 70/20/10 ethos where relevant: for strong usage-driven recommendations, phrase reasons around applicability ("this is a hands-on module you can apply directly") rather than pure topic match
- [ ] Example pattern: *"Because you told us you want to improve cash flow management and completed Intro to Bookkeeping, we suggest Cash Flow Forecasting for Small Businesses."*
- [ ] Keep LLM out of this by default — templated strings are more debuggable, and you'll need to justify this choice in the writeup anyway (see Phase 5)

### Phase 4 — API (45–60 min)

- [ ] `GET /users/{user_id}/recommendations?n=5` returning JSON: ranked position, score, reason, **and the per-component score breakdown** (even if not surfaced to end users, it makes live debugging trivial in the interview)
- [ ] FastAPI or Flask (Python) — pick whichever you're fastest in; stack choice isn't being tested
- [ ] Keep payloads lean — a nod to AMI's real mobile-first, sometimes low-connectivity user base; mention this as a deliberate constraint in the writeup, not just a generic "REST API" note

### Phase 5 — WRITEUP.md (45–60 min) — this is where AMI-specific alignment pays off most

Structure exactly per their brief, but weight each section toward AMI's real context:

1. **Approach & tradeoffs** — explain the pluggable-scorer architecture and why; where you deliberately kept it simple (e.g. no real collaborative filtering — usage data too sparse at this scale to trust; content similarity + cohort popularity instead)
2. **Signal weighting** — state starting weights plainly, and how you'd tune them (e.g. against a labeled "recovered hidden interest" metric from your synthetic data)
3. **Cold-start** — the continuous `usage_confidence` blending function, and its limits (e.g. purely heuristic work-context signal until real usage arrives)
4. **Measuring success** — anchor to AMI's real evaluation culture, not a generic completion-rate test:
   - Hypothesis: better-targeted recommendations increase **downstream practical application / self-reported business-performance improvement**, not just course completion
   - Metric: something in the spirit of their existing 86%-improved-performance tracking — e.g. % of recommended-course completers who report applying the tool in their business within 30 days
   - Method: holdout arm (cohort-popularity-only recommendations) vs. full-engine arm; explicitly address selection bias (more-engaged users are both more likely to be shown strong recommendations and more likely to complete anything — control for baseline engagement)
5. **Scaling to 10k+ users** — ground this in AMI's actual constraints, not generic SaaS scaling:
   - Precompute/batch recommendations nightly rather than scoring live per-request
   - Cache cohort-popularity aggregates (cheap to compute, expensive to do per-request at scale)
   - Keep API responses small for low-bandwidth mobile clients
   - Note what breaks first (likely: content-similarity computation over a growing catalog) and what you'd do next (approximate nearest-neighbor index over course tag vectors)

**LLM justification (bonus section, only if you have time):**
- [ ] Where it fits: wrapping the recommendation in a warmer, coach-like tone consistent with AMI's human-coaching model — and possibly semantic matching on free-text survey answers
- [ ] Where it should stay out: the core ranking/scoring logic — because a company reporting verified 86% business-performance improvement to funders and partners needs recommendation logic it can audit and defend, not something that can silently drift
- [ ] This directly answers one of their three stated live-interview questions — write it well

### Phase 7.5 — Minimal UI (bonus, 20–30 min max) — NOT a separate app

The brief lists this under explicitly-optional bonus items and says they'd rather see excellent core logic than a thin version of everything. Treat it accordingly:

- [ ] **One static HTML file, vanilla JS, zero build step** — e.g. `/ui/index.html` in the same repo, `fetch()`-ing your own `GET /users/{id}/recommendations` endpoint and rendering ranked position, score, and reason in plain DOM/template literals
- [ ] No React, no npm project, no separate repo, no build pipeline — this is a deliberate judgment call, not a shortcut: a separate frontend app trades hours away from the scoring engine/writeup (what's actually being evaluated) for a part they explicitly deprioritized
- [ ] Serve it as a static file from the same backend process (e.g. FastAPI `StaticFiles` mount) so `GET /` just works — one process, one README command, and you sidestep configuring CORS for a bonus feature
- [ ] Optional polish, still zero-build: a CDN `<script>` for Pico.css/Tailwind CDN for styling, or htmx/Alpine.js via CDN if you want a dropdown to switch between your 3 sample users — same single file, no tooling added
- [ ] In README, note explicitly *why* the UI is minimal ("kept to a single static page since the brief weights recommendation quality over frontend polish") — this turns a shortcut into a visible piece of judgment
- [ ] Build this **last**, only after the engine, API, and writeup are solid — if you're tight on time, cut this before you cut anything else
